# v0.2 — From demo to product

**Date:** 2026-05-26
**Scope:** Frontend UX polish, run analytics enrichment, schema additions for stored peak stats.

## Goal

Take DynoRun from MVP-shaped ("works, looks dev-ish") to product-shaped ("looks like something a tuner would screenshot"). Every screen surfaces a number the user cares about, in units they think in, with copy that respects their time.

## Non-goals

- Changing the physics pipeline (PIPELINE_VERSION stays at 2)
- Native-only features (Capacitor Haptics plugin — we use `navigator.vibrate` only)
- Service workers / offline / PWA install promotion
- Image uploads for vehicles
- Light theme / theme switcher
- Public share links / leaderboards
- Absolute power corrections (aero/drivetrain losses) — still wheel-power comparative

## Design summary

### A. Brand & shared design foundation

Build the primitives the rest of the work consumes.

**A1. Brand mark.** A single SVG dyno-needle logo as a React component (`<BrandLogo>`), used in `AppShell` header and `LoginScreen`. Favicon points to a static SVG with the same mark. Web app manifest updated with name "DynoRun", theme `#09090b`, accent `#f59e0b`.

**A2. Shared formatters.** `src/shared/format-time.ts` (relative + short absolute), `src/shared/format-power.ts` (handles kW / hp / PS conversion + formatting). Pure functions, unit-tested.

**A3. Units preference.** `src/app/units-context.tsx` exposes a React Context with `unit: 'kW' | 'hp' | 'PS'` and a setter. Persisted in `localStorage` under `dynorun:units`. Default `'kW'`. Wrap the app at `<App>` level so every screen can call `useUnits()`.

**A4. Extracted components/modules.**
- `src/ui/components/gps-warmup-card.tsx` — the GPS-lock progress card currently duplicated in `live-run-screen.tsx` and `calibration-step-measure.tsx`. Takes `{ accuracy_m, quality, fix_rate_hz, current_speed_kmh, warmup_started_at, good_since }` and renders the same UI both consumers had inline.
- `src/sensors/motion-fusion.ts` — extract the `FusionState` + `adaptAxis` + the live-speed-from-motion code currently inline in `calibration-step-measure.tsx` into a pure module with `createFusionState()`, `onGpsFix(state, sample)`, `onMotionEvent(state, event)`. Unit-tested.

### B. Schema: stored peak stats per run

The garage and vehicle-detail screens need per-run peak power without loading every curve. Add additive columns to `runs`:

```sql
ALTER TABLE runs ADD COLUMN title text;
ALTER TABLE runs ADD COLUMN peak_power_kw real;
ALTER TABLE runs ADD COLUMN peak_torque_nm real;
ALTER TABLE runs ADD COLUMN peak_power_rpm real;
```

All nullable — backfilled lazily when `ensureCurrentCurve` runs (which already happens on review/compare). New runs populate them at the end of `RunController.finishRun` after analysis.

Drizzle source of truth in `server/src/schema.ts`. Migration is purely additive; `drizzle-kit push` applies cleanly on deploy.

`Run` type in `src/shared/types.ts` extended with the four new optional fields. `runs` route's PATCH handler accepts `title`, `peak_power_kw`, `peak_torque_nm`, `peak_power_rpm` (existing `updateNotes` becomes `update`).

### C. Run review — analytics-rich

**C1. Stats row (2x2 grid replacing the current 1x2):**
- Peak power (with kW + secondary hp/PS based on units pref)
- Peak torque + RPM (newly displayed)
- Power band (RPM range where wheel_power_kw ≥ 80% of peak)
- Delta vs previous best run on the same vehicle (signed, e.g. "+8 kW")

**C2. Chart toggle.** Above the chart: a 3-state segmented control (Power / Torque / Both). "Both" overlays power left axis + torque right axis. Implementation in `PowerCurveChart` via a `mode` prop and a `pointsTransform` that picks the field. Colors use the app palette (amber for primary, cyan for secondary).

**C3. Editable run title.** New input above notes ("3rd · 14:32" default if blank). Saved to `runs.title` via PATCH.

**C4. Share / export.**
- "Export CSV" downloads `rpm,power_kw,torque_nm` for the current curve.
- "Share image" uses the Web Share API when present (mobile Safari/Chrome), falling back to download. Uses `<canvas>` to compose the chart + peak stats card as a 1080×1080 PNG. (No backend; everything client-side.)

### D. Compare — themed and useful

**D1. Run picker labels** become `${gear} · ${formatRelativeTime(started_at)} · ${peakKw} kW` — always populated, regardless of notes. Title (if set) shown on a second line.

**D2. Chart palette.** Replace matplotlib defaults with `['#f59e0b','#22d3ee','#a78bfa','#34d399','#f472b6','#fbbf24','#60a5fa']` (amber, cyan, violet, emerald, pink, gold, sky). Best run highlighted with a slightly thicker stroke (width 3 vs 2) and an `★` prefix in legend.

**D3. Power / Torque toggle** identical to review.

### E. Vehicle detail + garage — show off

**E1. Vehicle card subline** in garage: `{vehicle stats}` → `${count} runs · best ${peakKw} kW` (or "No runs yet" if empty). Uses `runs.peak_power_kw` directly — no curve fetches.

**E2. Empty garage onboarding.** First-visit card explaining the 3-step flow: "Add vehicle → Calibrate a gear → Drive and record."

**E3. Vehicle detail header gets a stats row:** total runs, best peak power (with units pref applied), most recent run relative time.

**E4. Run list rows** show `${peak_power_kw} kW (or hp/PS)` and relative time instead of ISO timestamp.

### F. Live run — race-ready (low-risk)

**F1. Optional countdown.** Settings has a toggle "Countdown before start" (default off). When on, tapping Start triggers a 3-2-1 overlay before the run actually begins recording.

**F2. Haptic on start/stop.** `src/app/haptics.ts` exposes `pulseStart()` (one 50ms) and `pulseStop()` (three 50ms). Uses `navigator.vibrate` if available; no-op otherwise. Called from `LiveRunScreen.startRun()` and the auto-stop callback path.

**F3. Live peak power estimate** during recording. The controller already emits live samples — we compute a small windowed `F=ma·v` on the client (5-point rolling) in `LiveRunScreen` and display it under RPM. (We do NOT change the offline pipeline; this is purely for live feedback.)

### G. Calibration — friendlier gear input

**G1. Gear chip picker.** Replace the text input with a row of chips: 2nd / 3rd / 4th / 5th / 6th / Custom. Selecting a numbered chip sets `gear_label` directly. Custom expands a text field.

**G2. RPM stepper.** Replace the raw input with a stepper: `-250 | 3000 | +250` (still allows direct text input above the stepper). Default 3000.

### H. Settings — account + units + cleanup

**H1. Account section** (top of screen): user email + sign-out button. `signOut()` already exists on `useAuth()`; after sign-out, redirect to `/login`.

**H2. Units preference** segmented control: kW / hp / PS. Writes to the new context.

**H3. Countdown toggle** for live run F1.

**H4. Developer section** at the bottom: existing Replay mode + a link to `/recordings`. Removed from the bottom nav.

### I. Bottom navigation rework

`AppShell` nav: **Garage / Runs / Settings** (3 tabs).

**I1. New Runs tab.** `src/ui/runs/all-runs-screen.tsx` lists every complete run across vehicles, newest first. Each row: vehicle name, gear, relative time, peak power (units-aware). Tapping opens the run review.

**I2. Recordings removed from nav** but `/recordings` route remains accessible from Settings → Developer.

### J. Login redesign

Logo + tagline "Your phone is a dyno." + amber primary button (matching the app palette). Same magic-link flow.

## Data flow

- **Peak stats write:** `RunController.finishRun()` computes the AnalyzedRun (already does), then calls `runRepository.update(id, { peak_power_kw, peak_torque_nm, peak_power_rpm })` immediately before `derivedCurveRepository.upsert`. Idempotent on re-analysis (`re-analyze.ts` mirrors the call).
- **Peak stats read:** Garage and vehicle-detail screens read directly from `Run.peak_power_kw`. No new endpoint; existing `GET /api/vehicles/:vehicleId/runs` returns the columns.
- **Units pref:** `<UnitsProvider>` wraps the app inside `<AuthProvider>`. Hook `useUnits()` returns `{ unit, setUnit, format }` where `format(kw)` produces the user-preferred display value+suffix.

## Error handling

- Sign-out failure → toast error, stay on screen
- Schema columns nullable → all UI must tolerate `null` peak stats and display "—" (e.g., legacy runs before backfill)
- `navigator.vibrate` may be undefined or no-op → call it inside a `if ('vibrate' in navigator)` guard
- Share API unavailable → fall back to PNG download
- Recordings PNG generation failure → fall back to CSV alone

## Testing

- `format-time`: relative buckets (just now, X min, X h, X d, then absolute short date)
- `format-power`: kW→hp (×1.341), kW→PS (×1.3596), formatting rules
- `motion-fusion`: integration accumulates correctly; axis adapts after GPS delta
- All existing 57 tests must remain green
- Manual smoke: run typecheck + build; load app in dev; verify each screen renders without thrown errors

## Risks & mitigations

- **R1 — Parallel agent file collisions.** Mitigation: Phase 1 (foundation + schema) lands in a single commit by the main session before any parallel agents start. Phase 2 agents each own a disjoint file set (matrix below).
- **R2 — Schema additive in dev but unmigrated locally.** Mitigation: don't require the columns at the API layer — return `null` if absent. Production deploy applies the migration automatically.
- **R3 — Units context not threaded everywhere.** Mitigation: define a single `formatPowerWithUnits(kw, unit)` helper, use it explicitly in each screen that displays power. Don't try to be too clever with global formatting.
- **R4 — Extracting `motion-fusion` breaks the working calibration measure flow.** Mitigation: extract to a module first with no behavior change, write tests that pin the existing behavior, then swap the inline call site.

## Open questions resolved

- **Run title vs. notes.** Title = short label (default auto-generated, editable). Notes = freeform.
- **Best run highlight criteria.** Highest `peak_power_kw` across complete runs for the vehicle. Ties broken by most recent.
- **Units pref applies to compare chart axis?** Yes — chart axis label becomes "Power (hp)" / "Torque (Nm)" / etc.
- **Where does the existing live "current speed" display switch sources during warmup vs recording?** Doesn't change. Live peak power estimator is purely additive UI inside `live-run-screen.tsx`.
